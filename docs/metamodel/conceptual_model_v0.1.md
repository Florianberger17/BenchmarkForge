# BenchmarkForge Metamodel v0.1

## Overview

BenchmarkForge defines a metamodel for generating reproducible benchmark tasks in the domain of enterprise data integration and ERP migration.

The benchmark generation process separates:

- Business context (Scenario)
- Data structures (Schema)
- Clean reference data (Clean Dataset)
- Benchmark input data (Benchmark Dataset)
- Data quality degradations (Noise Profile)
- Business logic (Business Rule)
- Benchmark challenges (Task)
- Expected solutions (Ground Truth)

Ground Truth is generated by applying Business Rules to Clean Datasets and serves as the expected solution for a specific Task.

---

## Core Entities

### Scenario

Defines the business context of a benchmark.

Examples:

- ERP Customer Migration
- ERP Vendor Migration
- Legacy System Archiving

A Scenario contains one or more Tasks.

---

### Schema

Defines the structure of source and target business objects.

Two schema types exist:

- Source Schema
- Target Schema

---

### Clean Dataset

Represents synthetically generated reference data.

Characteristics:

- Correct
- Complete
- Consistent
- Noise-free

The Clean Dataset represents the factual truth of a scenario.

---

### Noise Profile

Defines controlled data quality degradations.

Examples:

- Typographical errors
- Missing values
- Duplicate records
- Outdated information

---

### Benchmark Dataset

Represents the benchmark input provided to participants or AI systems.

Generated by applying a Noise Profile to a Clean Dataset.

---

### Business Rule

Represents domain-specific migration and transformation logic.

Examples:

- Migrate customers active within the last 3 years
- Remove duplicates
- Select most recent customer record

Business Rules are used to derive Ground Truth.

---

### Task

Represents a concrete benchmark challenge.

Examples:

- Schema Mapping
- Duplicate Detection
- Customer Migration
- Data Quality Improvement

Tasks belong to a Scenario.

Tasks operate on Benchmark Datasets.

Tasks are governed by Business Rules.

---

### Ground Truth

Represents the expected correct solution of a Task.

Ground Truth is generated by applying Business Rules to the Clean Dataset.

Ground Truth serves as the reference solution during evaluation.

## Design Principles

- Reproducibility
- Traceability
- Explicit separation of clean and noisy data
- Explicit representation of business logic
- Extensibility for future benchmark domains
